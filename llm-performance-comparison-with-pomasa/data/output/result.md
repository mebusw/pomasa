# 同一个任务，四个大模型：谁更适合做你的"AI总指挥"？

> 当我们谈论AI智能体时，我们到底在谈论什么？是工具的升级，还是认知方式的根本转变？

## 前言：一个真实的业务场景

想象这样一个场景：你是某电子企业的战略分析师，老板让你评估公司在2026年的政策风险。

这不是一个简单的搜索任务——你需要：
- 从宏观政策（"十五五"规划、中央经济工作会议）
- 到行业趋势（智能驾驶、L3商业化）
- 到企业自身业务板块
- 最终生成一份有数据、有洞察、可导出PDF的研究报告

如果你让AI来做这件事，你期待的是什么？

不是简单的问答，而是**系统的分析**。不是零散的信息，而是**结构化的报告**。不是泛泛而谈，而是**有数据、有来源、有洞察的深度研究**。

这正是**多智能体系统（MAS）**要解决的问题——让多个专门的AI智能体分工协作，共同完成复杂任务。

但问题来了：**谁来做这个"总指挥"？**

这个"总指挥"——我们称之为**编排器（Orchestrator）**——决定了整个系统的成败。它需要理解任务目标、分解工作步骤、调用合适智能体、检查质量、整合结果。

**本次评测的目的很简单：用同一个实际任务，看看不同大模型生成的"总指挥"，表现会有多大差别。**

---

## 关于评测框架：POMASA

为了进行这次公平的对比，我们需要一个统一的框架。恰好，**POMASA**是一个非常适合的选择。

### 为什么选择POMASA？

简单说：**它让不同模型站在同一起跑线上。**

POMASA不是某家公司的专属工具，而是一套**模式语言**——包含18个相互关联的设计模式，覆盖了多智能体系统的核心问题：

- **任务应该如何分解？**（COR核心模式）
- **知识应该如何组织？**（STR结构模式）
- **协作应该如何进行？**（BHV行为模式）
- **质量应该如何保障？**（QUA质量模式）

更重要的是，**POMASA Generator**可以根据你的意图，自动生成符合这些模式的编排器和智能体。这样，我们就可以让不同模型基于同一套模式目录，生成各自的"总指挥"，然后公平地比较它们的实际表现。

### 完全不用写代码

POMASA还有一个优势：**完全不用写代码**。

你不需要学习Python、理解LangChain/AutoGen、编写复杂的编排逻辑。你只需要用自然语言描述任务，选择需要的模式（比如要导出PDF吗？需要质量检查吗？），Generator就会生成完整的系统。

这让评测更加纯粹——我们比较的是模型的理解和推理能力，而不是编程能力。

## 一、评测设计：同一个任务，四个模型

### 1.1 任务设定

我们选择了一个真实的复杂任务：**分析某电子企业在2026年的政策风险与机遇**。

任务要求：
- 从宏观政策（"十五五"规划、中央经济工作会议）
- 到行业趋势（智能驾驶、L3商业化）
- 到企业分析（该企业的三大业务板块）
- 最终生成一份结构化的研究报告
- 并且需要导出为DOCX/PDF格式

### 1.2 参赛模型

我们选择了四个当前最顶尖的大模型：

| 模型 | 特点 | 搜索能力 |
|------|------|---------|
| **Claude Opus 4** | Anthropic最强模型，以逻辑严谨著称 | 无 |
| **GLM-4.7** | 智谱AI旗舰模型，中文能力优秀 | 无 |
| **GLM-4.7 + Search** | 同上，但增强了Google搜索能力 | Google Search |
| **Gemini 3 Pro** | Google最新模型，多模态能力强 | 无 |
| **Gemini 2.5 Flash** | Google轻量级模型，速度快 | 无 |

### 1.3 评测维度

我们从两个角度进行评测：

**编排器生成质量**
- 结构完整性：是否有清晰的工作流程？
- 模式遵循度：是否正确应用了POMASA模式？
- 可执行性：生成的编排器能否真正运行？
- 语言一致性：是否遵循了用户的语言要求？

**执行结果质量**
- **高（Height）**：战略高度，是否站在决策者视角？
- **广（Width）**：覆盖广度，信息是否全面？
- **深（Depth）**：分析深度，洞察是否独到？
- **真（Truth）**：数据准确性，来源是否可靠？
- **善（Kindness）**：实用价值，建议是否具体？
- **美（Beauty）**：结构表达，报告是否易读？

### 3.2 关键差异分析

#### **语言遵循：最基础的考验**

用户明确要求：**Agent Blueprint Language = Chinese**

| 模型 | 结果 |
|------|------|
| Gemini 2.5 Flash | ❌ **全篇英文** |
| Gemini 3 Pro | ✅ 中文 |
| Claude Opus 4 | ✅ 中文 |
| GLM-4.7 | ✅ 中文 |

**洞察**：语言一致性是最基本的指令遵循能力。Gemini 2.5 Flash在这一步就失败了，这意味着在实际执行中，Agent会使用错误的指令语言，导致整个系统运行异常。

#### **STR-09（导出管道）：细节见真章**

用户明确要求：生成报告后需要导出为DOCX/PDF格式。

| 模型 | 结果 |
|------|------|
| Gemini 2.5 Flash | ❌ 遗漏 |
| Gemini 3 Pro | ❌ 遗漏 |
| Claude Opus 4 | ✅ 完整包含（设立"阶段五"执行export.sh） |
| GLM-4.7 | ✅ 完整包含（设立"阶段六"执行export.sh） |

**洞察**：这是模式遵循能力的直接体现。只有Claude和GLM正确识别并应用了STR-09模式（交付物导出管道），在流程末尾增加了导出步骤。Gemini系列都忽略了这一明确需求。

#### **BHV-02（忠实Agent实例化）：核心中的核心**

这是多智能体系统正确执行的关键：**如何调用子Agent？**

错误方式：`run_agent("policy_analyst", "分析宏观政策")`
正确方式：`请阅读 agents/policy_analyst.md 并严格按照该Blueprint执行`

| 模型 | BHV-02遵循度 |
|------|-------------|
| Gemini 2.5 Flash | ❌ 使用简化的run_agent |
| Gemini 3 Pro | ⚠️ 仅提及"调用Agent"，未强化约束 |
| Claude Opus 4 | ✅ 明确指令"阅读agents/xx.md" |
| **GLM-4.7** | ✅✓ **专门章节强调"Agent调用标准"，并给出细则** |

**GLM-4.7的亮点**：
```
## Agent调用标准
### 必须遵守的原则
1. 不总结或改写Blueprint（BHV-02）
2. 每个独立任务启动独立的子代理
3. 标准调用方法：
   请阅读 agents/XX.xxx.md 并严格按照该Blueprint执行
   参数：
   - PARAM1: value1
4. 验证子代理结果
5. 边界约束明确
```

**洞察**：只有GLM-4.7明确强调"不总结或改写Blueprint"，这是防止信息传递损耗的关键。在实际运行中，如果Orchestrator试图自己概括任务传给子Agent，就会违反BHV-02，导致子Agent无法获得完整的上下文信息。

#### **验证流程：工程化思维的体现**

| 模型 | 验证机制 |
|------|---------|
| Gemini 2.5 Flash | ❌ 无 |
| Gemini 3 Pro | ⚠️ 隐式，仅在Review阶段提及 |
| Claude Opus 4 | ✅ 专门章节列出QUA-03、QUA-05等 |
| **GLM-4.7** | ✅✓ **每阶段都有详细"完成标准验证"小节** |

**GLM-4.7的示例**：
```
## 阶段一：初始化与数据收集
### 完成标准验证
- [ ] data/01.materials/overview.md 存在且非空
- [ ] data/01.materials/topic_*.md 至少3个主题文件
- [ ] 所有文件符合命名规范
```

**洞察**：GLM-4.7展现了极强的工程化思维——它在"初始化"阶段直接给出了`mkdir -p ...`命令，这在实际工程中非常实用。每个阶段后的验证清单确保了质量可控。

### 2.3 模式遵循情况对比

| 模式 | Gemini 2.5 | Gemini 3 Pro | Claude Opus 4 | GLM-4.7 |
|------|-----------|-------------|---------------|---------|
| BHV-01 编排管道 | ✓ | ✓ | ✓ | ✓ |
| **BHV-02 忠实实例化** | ✗ | 部分 | 部分 | ✓✓ |

**关键发现**：

- **Gemini的路径**像是一位"国际分析师"——不局限于中文信息，更看重全球资本市场和前沿技术标准（如UNECE的标准）对国内的影响，独家访问了大量英文学术、国际金融及国际标准化组织网站

### 4.4 核心洞察

**搜索能力 = 信息广度 × 数据准确性 × 分析深度**

- 信息广度：报告篇幅1183行，覆盖政策、行业、企业全链路
- 数据准确性：每个数据都有完整来源链接
- 分析深度：10大机遇、8大风险的系统化分析

**对比基础版GLM-4.7**：
- 篇幅：353行 → 1183行（**3.35倍**）
- 评分：7.5/10 → 9.5/10（**提升26.7%**）
- 风格：从"政策解读员"升级为"全面行业分析师"

**结论**：在多智能体系统中，**工具（Tools）的作用是决定性的**。没有实时信息/外部知识库的注入，Agent只能生产平庸的内容。

## 五、多智能体编排能力洞察

通过对比五份报告，我们可以反推各模型的**Agent编排（Orchestration）**特点。

### 5.1 Gemini-3-Pro："金字塔原理"

推测其Agent工作流采用了**"结论先行，论据支撑"**的模式。

- Final Report Generator智能体具有很强的**综合（Synthesis）**能力
- 能够大刀阔斧地删减冗余信息，只保留最核心的战略判断
- 77行报告传达了最高密度的战略价值

**体现**：
- 篇幅最短（77行），但战略价值最高
- "地缘政治是核心挑战"的洞察极具穿透力
- NTT合作等建议具体且独特

### 5.2 Claude-Opus-4："线性推导"

推测其Agent工作流是严密的**逻辑链条**：宏观 → 行业 → 企业。

- 每个环节的Agent都非常忠实地完成了本职工作
- 最终报告是各环节高质量输出的完美拼接
- 数据一致性极强（如该企业营收276.18亿元在文中多次出现且未矛盾）

**体现**：
- 结构最为严谨，宛如标准的深度行业研报
- 各章节之间过渡自然
- 显示出极强的长窗口一致性

### 5.3 GLM-4.7 (+Search)："发散-收敛"

推测其Agent被鼓励**广泛搜集信息**。

- Search MCP让Data Collector智能体极其活跃

- 最终报告呈现出"百科全书"式的详尽，但在"收敛"提炼上稍显不足

**体现**：
- 报告篇幅最长（1183行）
- 信息覆盖面最广，包含政策协同性分析、不确定性分析等独特内容

## 七、POMASA：零代码创建智能体系统

在本文的最后，我想回到最开始的问题：**为什么选择POMASA？**

### 7.1 完全不用写代码

POMASA最大的优势是：**完全不用写代码**。

你不需要：
- ❌ 学习Python
- ❌ 理解LangChain/AutoGen等框架
- ❌ 编写复杂的编排逻辑
- ❌ 调试API调用

你只需要：
- ✅ 用自然语言描述你的意图
- ✅ 选择你需要的模式（要导出PDF吗？需要质量检查吗？）
- ✅ 让POMASA Generator生成编排器和智能体

### 7.2 比Claude Code的Skill更容易

Claude Code的Skill需要：
1. 编写Skill定义文件（JSON/YAML）
2. 编写Prompt模板
3. 配置工具调用
4. 测试和调试

POMASA只需要：
1. 告诉Generator你的意图
2. 选择你需要的模式
3. 生成的每个Agent就是一个**SKILL**

**重点是**：POMASA生成的每个智能体都是从零开始生成的，没有预设模板。它们可以：
- 单独使用（比如只调用"宏观政策分析师"）
- 组合使用（比如调用"宏观政策分析师" + "行业分析师"）
- 复用扩展（比如把"宏观政策分析师"复制到另一个项目中）

### 7.3 站在认识论高度

POMASA不是在解决"如何让AI干更多活"的问题，而是在回答：
- **任务应该如何分解？**（COR核心模式）
- **知识应该如何组织？**（STR结构模式）
- **协作应该如何进行？**（BHV行为模式）
- **质量应该如何保障？**（QUA质量模式）

这是**认识论**的视角，而不是**工程实现**的技巧。

### 7.4 立即上手

POMASA的核心理念是：**让AI理解并执行你的认识论框架**。

你只需要：
1. 准备你的模式目录（Pattern Catalog）
2. 告诉Generator你的意图和需求
3. 选择你需要的模式
4. 生成符合你需求的编排器和智能体

然后，你就可以：
- 让Orchestrator协调整个流程
- 让每个Agent专注于自己的职责
- 通过文件管道透明地查看中间结果
- 根据需要优化和调整

**最重要的是**：所有这些都是用自然语言完成的，没有一行代码。

## 八、结语：正视差距，也要看到希望

本次评测揭示了一个核心洞察：**同一个框架和机制，不同模型、不同搜索引擎带来的表现差别是巨大的**。

- **编排器生成**：GLM-4.7和Claude Opus 4在模式遵循和工程执行上明显优于Gemini系列
- **执行质量**：GLM-4.7 + Search在信息广度、数据准确性、分析深度上全面领先
- **搜索引擎**：搜索能力的作用是决定性的，能让系统产生质的飞跃
- **模型选择**：应根据具体场景选择，而不是盲目追求"最强模型"

### 正视差距

我们需要承认，在本次评测中：
- **Gemini系列**在模式遵循和工程细节上表现较弱
- **国内模型**（GLM-4.7）在编排器生成上表现优异，但在战略洞察上与Gemini 3 Pro仍有差距

### 看到希望

但更重要的是，我们也看到了**民族工业的希望**：

1. **GLM-4.7在编排器生成上全面领先**：在模式遵循、工程执行、验证流程等方面，GLM-4.7表现最优。要知道GLM-4.7对标的仅仅是中档的Claude Sonnect-4.5而并非其最强模型Opus说明国产模型在理解复杂框架、生成可执行代码方面已经具备国际竞争力

2. **搜索能力的决定性作用**：GLM-4.7 + GoogleSearch的评分提升26.7%，说明**工具生态**可能比模型本身更重要。国产模型可以通过构建强大的工具生态来弥补模型能力的差距

3. **场景化选择的重要性**：没有绝对"最强"的模型，只有"最适合"的模型。国产模型在特定场景（如中文政策分析、行业研究）上已经具备独特优势

### 未来可期

POMASA的价值在于：**它让你能够用认识论的方式思考问题，让AI来处理工程实现的细节**。

这不是工具的升级，而是认知方式的根本转变。

而随着国产模型和工具生态的不断发展，我们有理由相信：
- 在多智能体编排领域，国产模型已经具备国际竞争力
- 通过框架和工具的加持，国产模型可以在特定场景下超越国际顶尖模型
- 在AI应用落地的道路上，我们走出了自己的道路

**你准备好用AI的方式来构建AI系统了吗？**
